### Finding 1: Test file path for Task 11 is imprecise

- **Category**: Weakness
- **Severity**: IMPORTANT
- **Evidence**: Test Plan table, line 734:
  ```
  | Task 11: PokemonResult DTO + formatter | Unit | tests/Unit/Core/Component/PokeParser/Domain/... (or standalone) | ... |
  ```
- **Problem found**: The Test Plan maps Task 11 (PokemonResult DTO + formatter) to a test path under `Domain/`. The `PokemonResult` DTO lives in `Application/Dto/` (`src/Core/Component/PokeParser/Application/Dto/PokemonResult.php`) and `PokemonResultFormatter` lives in `Presentation/Cli/` (`src/Presentation/Cli/PokemonResultFormatter.php`). Neither belongs under `Domain/`. The `(or standalone)` parenthetical leaves the developer to guess the correct path.
- **Why it matters**: A developer following the plan literally would place tests in the wrong directory, creating a namespace mismatch with the PSR-4 autoload-dev mapping (`Gete\PokeParser\Test\` → `tests/`). The test for `PokemonResult` should be at `tests/Unit/Core/Component/PokeParser/Application/Dto/PokemonResultTest.php` and the formatter test at `tests/Unit/Core/Component/PokeParser/Presentation/Cli/PokemonResultFormatterTest.php`.
- **Recommendation**: Replace the imprecise path with two explicit entries in the Test Plan table:
  - `tests/Unit/Core/Component/PokeParser/Application/Dto/PokemonResultTest.php` — verifies DTO construction and property access
  - `tests/Unit/Core/Component/PokeParser/Presentation/Cli/PokemonResultFormatterTest.php` — verifies byte-identical output format
- **Workflow gap**: Cross-artefact drift — the directory tree (lines 67–110) correctly places these files, but the Test Plan table (line 734) uses a different path. This is a recurring class of defect: a structural assertion in one section of the plan (directory tree) is contradicted by a reference in another section (test plan). A rule should exist: "Every test file path in the Test Plan table must match a source file path in the Directory Tree or be explicitly justified as a new test-only path." This rule should be added to the `telamon.plan_implementation` skill's Pre-FINISHED Hygiene Gate checklist.

**What would make me comfortable with this approach**: Two explicit test file paths in the Test Plan table, one for the DTO test and one for the formatter test, both matching the PSR-4 namespace convention.

### Finding 2: Step 16 fixture loading mechanism is underspecified

- **Category**: Suggestion
- **Severity**: MINOR
- **Evidence**: Step 16, lines 712–716:
  ```
  Fixture approach: record the PokeAPI responses for the four Pokemon and their growth rates
  as JSON files in tests/Fixtures/. Create a fake HTTP client or adapter that returns these fixtures.
  ```
- **Problem found**: The plan specifies *where* fixtures live (`tests/Fixtures/`) and *what* they contain (recorded PokeAPI responses), but does not specify *how* the fake adapter is constructed or *how* fixtures are loaded. The developer must decide between:
  - A hand-written fake implementing the port interface that reads JSON files
  - A PHPUnit data provider that loads fixtures
  - A test helper trait
  - Injecting fixtures via constructor or setter
- **Why it matters**: Minor — an experienced developer can implement this without ambiguity. However, the fixture-loading mechanism affects test maintainability (e.g., if the fixture file format changes, all tests break). Specifying the pattern upfront prevents divergent implementations.
- **Recommendation**: Add a brief note specifying the fixture-loading pattern. Suggested approach: a hand-written fake adapter (e.g., `FakePokeApiPokemonAdapter` in `tests/Support/`) that accepts fixture data in its constructor and implements the port interface. The approval test instantiates the fake with known fixture data and wires it into the pipeline. This avoids Mockery's `readonly class` limitation and keeps the test self-contained.

**What would make me comfortable with this approach**: A one-sentence fixture-loading strategy in Step 16's Notes section, e.g., "Use a hand-written fake adapter under `tests/Support/` that accepts fixture data in its constructor and implements the port interface."

### Finding 3: Step 15 dispatch-counting mechanism is underspecified

- **Category**: Suggestion
- **Severity**: MINOR
- **Evidence**: Step 15, line 688:
  ```
  - Verify correct number of sub-query dispatches (one per Pokemon per query type).
  ```
- **Problem found**: The plan specifies *what* to verify (dispatch count) but not *how*. The developer must choose between:
  - Mockery spies on the bus's `dispatch` method
  - Counting calls to mock query handlers
  - A custom bus implementation that tracks dispatches
- **Why it matters**: Minor — the verification is achievable with any of these approaches. However, the choice affects test readability and coupling to the bus API (which is unknown until gate-bus-api).
- **Recommendation**: Add a brief note: "Use Mockery spies on the registered mock handlers to count invocations, or wrap the bus in a test double that tracks dispatch calls. Avoid coupling the test to the bus's internal dispatch mechanism."

**What would make me comfortable with this approach**: A one-sentence dispatch-counting strategy in Step 15's Notes section.

### Finding 4: Step 2 algorithm comment imprecision on initial "previous level"

- **Category**: Suggestion
- **Severity**: MINOR
- **Evidence**: Step 2, lines 218–223:
  ```
  // Algorithm: iterate levels sorted ascending by experience.
  // Track the previous entry's level.
  // When baseExperience < entry.experience, return the previous entry's level.
  // If no entry has experience > baseExperience, return the last entry's level.
  // If the first entry already has experience > baseExperience, return 0.
  ```
- **Problem found**: The algorithm description says "return the previous entry's level" without specifying the initial value of the "previous level" before the first iteration. In the original script, `$possibleLevel` starts at `0`, not at the first entry's level. The comment "If the first entry already has experience > baseExperience, return 0" correctly handles this case, but the main algorithm description could mislead a developer into thinking the "previous entry's level" for the first entry is the first entry's level (which would produce wrong results).
- **Why it matters**: Minor — the expected values in the plan (lines 227–233) make the correct behaviour unambiguous. A developer implementing from the expected values will get the right algorithm regardless of the comment imprecision.
- **Recommendation**: Add "Initialize previous level to 0" as the first bullet in the algorithm comment, before "Track the previous entry's level."

**What would make me comfortable with this approach**: The algorithm comment includes "Initialize previous level to 0" as its first step.

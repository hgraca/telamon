# ADR: Opencode Plugin Pattern — Thin Trigger Delegating to Underlying Tool

Created by: Herberto
Created time: May 17, 2026
Last edited by: Herberto
Last updated time: May 17, 2026

## Summary

**In context of** building opencode plugins that hook into session events,
**facing**
- the risk of plugins accumulating business logic and becoming hard to test or reuse independently
- the need for a consistent naming convention so plugin purpose is immediately readable from the filename
- the requirement (see [PDR: Opencode plugins](../PDRs/20260517000001-plugins.md)) that a plugin must be a trigger only and the underlying tool must exist before the plugin is created

**we decided for** a strict thin-trigger pattern — every plugin is a minimal event hook that immediately delegates to a pre-existing tool — **to achieve**
- testable tools that work independently of the opencode event system
- plugins that are trivially auditable (no hidden logic)
- filenames that encode both the triggering event and the delegated tool

**accepting** that creating a plugin always requires a tool to exist first, which adds one step to the workflow.

## Context

Telamon ships plugins under `src/instructions/plugins/`. Each plugin hooks into an opencode lifecycle
event (e.g. `session.idle`, `file.edited`, `tool.execute.before`) and performs some action. Without a
clear structural rule, plugins tend to grow: logic gets added inline, the plugin becomes the only
entry point for that capability, and the capability becomes untestable without running opencode.

The product rule (PDR linked in `see`) mandates three constraints:
1. A plugin is a trigger only — no business logic.
2. The underlying tool must exist before the plugin is created.
3. Plugin filenames follow `on_<event>-<tool-name>`.

## Options

### Option 1 — Thin trigger: plugin delegates entirely to tool (decided)

The plugin file contains only the hook wiring. All logic lives in a separate tool
(`src/instructions/tools/<tool-name>/`). The plugin spawns or imports the tool.

**Pros**
- Tool is independently testable from CLI
- Plugin is trivially auditable — one glance shows what it does
- Tool can be reused by other plugins or called directly by agents

**Cons**
- Requires creating the tool first (extra step, but enforced by PDR)

### Option 2 — Self-contained plugin

Plugin contains both the hook wiring and the business logic inline.

**Pros**
- Single file per capability

**Cons**
- Logic untestable without running opencode
- Violates PDR rule 1 (plugin must be trigger only)
- Capability cannot be reused by agents or other plugins

## Decision

We adopt **Option 1**: every plugin under `src/instructions/plugins/` is a thin trigger that
delegates to a pre-existing tool. The plugin file must not contain business logic.

### Naming convention

```
on_<event>-<tool-name>.<ext>
```

- `<event>` — the opencode hook name, with dots replaced by underscores
- `<tool-name>` — kebab-case name of the underlying tool

Examples:

| Plugin filename                         | Hook                  | Delegated tool     |
|-----------------------------------------|-----------------------|--------------------|
| `on_session_idle-remember-session.ts`   | `session.idle`        | `remember-session` |
| `on_file_create-qmd-refresh.ts`         | `file.edited`         | `qmd-refresh`      |
| `on_tool_execute_before-rtk-rewrite.ts` | `tool.execute.before` | `rtk` (rewrite)    |

### Plugin file structure

```typescript
import type { Plugin } from "@opencode-ai/plugin"
import path from "path"

// on_<event>-<tool-name>.ts
// Thin trigger — delegates all logic to <tool-name>.sh / <tool-name>.ts.
// No business logic here.

export const OnEventToolNamePlugin: Plugin = async ({ project, $ }) => {
  return {
    "<hook-name>": async (input, output) => {
      const script = path.join(import.meta.dir, "../tools/<tool-name>/<tool-name>.sh")
      try {
        await $`bash ${script}`.cwd(project.worktree).quiet()
      } catch (err) {
        // Graceful degradation — never throw from hook
      }
    },
  }
}
```

### Workflow: creating a plugin

1. **Check if the underlying tool exists** under `src/instructions/tools/<tool-name>/`.
2. **If tool does not exist** — create it first using the `create-tool` skill before proceeding.
3. **Create the plugin file** named `on_<event>-<tool-name>.<ext>` in `src/instructions/plugins/`.
4. **Plugin body** — hook wiring only; delegate immediately to the tool.
5. **Register** the plugin in `opencode.jsonc` if required.

### Rationale

1. **Testability** — the tool can be exercised with `bash <tool-name>.sh <args>` without starting
   opencode. The plugin is just wiring.
2. **Reusability** — agents can call the tool directly via MCP; the plugin is one consumer among
   potentially many.
3. **Auditability** — a plugin file that is only wiring is trivially reviewable. Logic hidden inside
   a plugin hook is invisible to static analysis and hard to test.
4. **Naming as documentation** — `on_session_idle-remember-session` tells you the event, the tool,
   and the relationship at a glance.

## Consequences and Follow-Up Work

- Apply this pattern to every new plugin added under `src/instructions/plugins/`.
- Audit existing plugins and align them to this pattern where they deviate.
- Update `src/instructions/skills/self-improvement/create-plugin/SKILL.md` to enforce the PDR rules
  and invoke the `create-tool` skill when the underlying tool does not yet exist.

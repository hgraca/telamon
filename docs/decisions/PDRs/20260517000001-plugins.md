## Opencode plugins

An opencode plugin (`src/instructions/plugins/`) must be a thin trigger for an underlying tool — it
contains no business logic of its own. All meaningful work lives in the tool the plugin delegates to.

### Rules

1. **Plugin is a trigger only.** A plugin must not contain business logic. It hooks into an opencode
   event and delegates immediately to an underlying tool (a bash script, a TypeScript MCP tool, or
   an external binary). The plugin file is the wiring; the tool file is the implementation.

2. **Tool first.** If the underlying tool does not yet exist, create the tool before creating the
   plugin. A plugin without a backing tool is incomplete and must not be committed.

3. **Naming convention.** Plugin file names must follow the pattern:

   ```
   on_<event>-<tool-name>
   ```

   Where `<event>` is the opencode hook name (underscores preserved) and `<tool-name>` is the
   kebab-case name of the underlying tool.

   Examples:
   - `on_file_create-qmd-refresh` — fires on `file.edited`, delegates to the `qmd-refresh` tool
   - `on_session_idle-remember-session` — fires on `session.idle`, delegates to the `remember-session` tool
   - `on_tool_execute_before-rtk-rewrite` — fires on `tool.execute.before`, delegates to the `rtk` tool
